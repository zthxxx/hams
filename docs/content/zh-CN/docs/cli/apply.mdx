# hams apply

把你 Hamsfile 里写下的所有资源，在当前这台机器上"兑现"。这是 hams 日常跑得最多的一条命令，
不管是新机器第一次铺环境，还是后面手动改了 YAML 想同步过去，都走它。

它干的事说穿了就三步：读 Hamsfile → 跟机器当前状态比对 → 差啥补啥、多啥卸啥、旧了就升。

## 用法

```bash
hams apply [flags]
```

不带任何 flag，就会把当前 store 里所有 Hamsfile 应用到本机。

## 场景示例

```bash
# 最常见：把当前 store 里的所有配置跑一遍
hams apply

# 新机器开箱：从 GitHub 仓库拉 store，直接开跑
hams apply --from-repo=zthxxx/hams-store

# 不放心？先 dry-run 预览一下会动哪些东西
hams apply --dry-run

# 只想跑 Homebrew 和 pnpm，别的先放一边
hams apply --only=homebrew,pnpm

# 除了 Ansible 其他都跑（Ansible 通常最耗时）
hams apply --except=ansible
```

## Flag

| Flag | 类型 | 说明 |
|------|------|------|
| `--from-repo` | string | 从远端 GitHub 仓库克隆 store 再 apply。适合新机器开荒。 |
| `--only` | string | 只跑指定的 provider，逗号分隔 |
| `--except` | string | 排除掉指定的 provider，其他都跑 |
| `--prune-orphans` | bool | 对那些只有 state 文件、没有对应 hamsfile 的 provider 做清理：把 state 里登记的所有资源都卸掉。**有破坏性，默认关闭。** 适合那种"我已经不用这个工具了，让 hams 帮我清理"的场景。注意：用之前确认 hamsfile 不是因为半截 git pull 缺失的，否则会一次性卸掉一堆 |
| `--dry-run` | bool | 只打印计划，不真动手 |

## 执行流程

一次 `apply` 背后 hams 干了这些事：

1. **加锁** — 给当前机器上一把全局锁，防止两个 apply 同时跑互相踩脚
2. **刷新状态** — 先对每个 provider 跑一遍 `refresh`，把机器现状写进 `.state/`
3. **算差异** — 拿 Hamsfile（你想要的样子）和 `.state/`（现在的样子）对比，算出要装、要卸、要改的资源清单
4. **按优先级执行** — 按照 `provider_priority` 的顺序一个个 provider 跑。默认 `bash` 走在最前，
   这样你能用 bash 步骤先把 Homebrew 这种依赖装上
5. **写回状态** — 每个资源装完就立刻更新对应的 `.state/<machine-id>/<Provider>.state.yaml`

中途要是 Ctrl+C 打断了也不怕，下次再跑 `apply` 会从断点继续 —— 已经处理完的资源不会重跑。

## 关于 sudo

如果你的 profile 里有 `apt` 这类需要提权的 provider，hams 会在启动时一次性问你密码，
然后每 4 分钟在后台自己刷 sudo 票据，保证整段 apply 过程不会再弹窗打扰你。

## 失败了怎么办

单个资源失败不会让整次 apply 都挂掉。hams 会接着跑剩下的，结束后在汇总里标出哪些失败了，
并给出退出码 `4`（部分失败）。日志全量保留在 `~/.local/share/hams/` 下，按月归档。
排查完再跑一次 `apply`，它只会重试失败的那些。

<Callout type="tip">
  第一次在陌生 store 上跑 apply，强烈建议先 `--dry-run` 看一眼清单，确认没问题再真跑。
</Callout>
