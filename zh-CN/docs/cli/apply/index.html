    hams apply

    把你 Hamsfile 里写下的所有资源，在当前这台机器上”兑现”。这是 hams 日常跑得最多的一条命令， 不管是新机器第一次铺环境，还是后面手动改了 YAML 想同步过去，都走它。

    它干的事说穿了就三步：读 Hamsfile → 跟机器当前状态比对 → 差啥补啥、多啥卸啥、旧了就升。

    用法

    hams apply [flags]

    不带任何 flag，就会把当前 store 里所有 Hamsfile 应用到本机。

    场景示例

    # 最常见：把当前 store 里的所有配置跑一遍
hams apply
 
# 新机器开箱：从 GitHub 仓库拉 store，直接开跑。
# 如果机器上还没装 Homebrew，加上 --bootstrap 让 hams 代你跑官方的
# install.sh（交互终端会先弹提示确认）
hams apply --bootstrap --from-repo=zthxxx/hams-store
 
# 不放心？先 dry-run 预览一下会动哪些东西
hams apply --dry-run
 
# 只想跑 Homebrew 和 pnpm，别的先放一边
hams apply --only=homebrew,pnpm
 
# 除了 Ansible 其他都跑（Ansible 通常最耗时）
hams apply --except=ansible

    Flag

    Flag类型说明
    --from-repostring从远端 GitHub 仓库克隆 store 再 apply。适合新机器开荒。
    --onlystring只跑指定的 provider，逗号分隔
    --exceptstring排除掉指定的 provider，其他都跑
    --no-refreshbool跳过 apply 前的 refresh 阶段。默认 apply 会先扫一遍机器状态捕获 drift；加了 --no-refresh 就直接信任 .state/ 里的数据去算 diff。适合刚跑完 hams refresh 想再 apply 一次的场景
    --prune-orphansbool对那些只有 state 文件、没有对应 hamsfile 的 provider 做清理：把 state 里登记的所有资源都卸掉。有破坏性，默认关闭。 适合那种”我已经不用这个工具了，让 hams 帮我清理”的场景。注意：用之前确认 hamsfile 不是因为半截 git pull 缺失的，否则会一次性卸掉一堆
    --bootstrapbool当某个 provider 的前置工具没装时（典型如 Homebrew），自动跑该 provider 声明的 bootstrap 脚本。会从网上跑 bash 脚本，所以要显式开启。默认关闭；交互终端下会改为 [y/N/s] 提示
    --no-bootstrapbool关闭交互终端下的 bootstrap 提示。CI 里如果你希望前置工具缺失时立刻失败、而不是挂起等你输入，用它
    --dry-runbool只打印计划，不真动手

    执行流程

    一次 apply 背后 hams 干了这些事：

    1. 加锁 — 给当前机器上一把全局锁，防止两个 apply 同时跑互相踩脚
    2. 刷新状态 — 先对每个 provider 跑一遍 refresh，把机器现状写进 .state/
    3. 算差异 — 拿 Hamsfile（你想要的样子）和 .state/（现在的样子）对比，算出要装、要卸、要改的资源清单
    4. 按优先级执行 — 按照 provider_priority 的顺序一个个 provider 跑。默认 bash 走在最前， 这样你能用 bash 步骤先把 Homebrew 这种依赖装上
    5. 写回状态 — 每个资源装完就立刻更新对应的 .state/<machine-id>/<Provider>.state.yaml

    中途要是 Ctrl+C 打断了也不怕，下次再跑 apply 会从断点继续 —— 已经处理完的资源不会重跑。

    关于 bootstrap 提示

    新机器上可能某个 provider 的前置工具还没装。目前有几个 provider 支持 bootstrap consent 流程 —— 各自的 install 脚本在 provider manifest 里声明：

    ProviderInstall 脚本宿主
    brew（Homebrew）curl -fsSL .../install.shmacOS / Linuxbrew
    pnpmnpm install -g pnpm前置需要 npm 已在 PATH
    dutibrew install dutimacOS；链式经过 brew
    masbrew install masmacOS；链式经过 brew
    ansiblepipx install --include-deps ansible前置需要 pipx（apt install pipx / brew install pipx）

    其他 provider（npm、cargo、goinstall、uv、code、apt、defaults、git）故意不接入 --bootstrap —— 语言 runtime 的安装属于你自己的决定（nvm / fnm / rustup / 发行版）， code 需要 GUI 应用，apt/defaults 平台门控，git 是 hams 安装器运行所需的前置。 这些 provider 前置缺失时 hams 直接给一条 “not found in PATH” 的简单错误 + 安装提示。

    当 hams 跑到一个缺前置工具的 provider 时，它会停下来，然后根据当前情境做三件事之一：

    1. 你传了 --bootstrap：直接通过 bash provider 执行 manifest 声明的 install 脚本，跑完再重试当前 provider。
    2. 交互终端且没传 --no-bootstrap：把即将执行的脚本原文打出来，提醒你可能要 sudo 密码、 macOS 上可能触发 Xcode CLT 对话框、可能被公司代理挡住，然后问 [y/N/s]（yes / no / 跳过这个 provider）。
    3. 其他情况（CI、管道、显式 --no-bootstrap）：hams 直接给一条可复制的错误信息后退出 —— 不会 发任何网络请求，不会对系统做任何改动。错误信息里包含二进制名、完整 install 脚本、以及 --bootstrap 这条 copy-paste 的恢复指令。

    hams 永远不会在没有你明确授权的情况下，从网上跑 install 脚本。这和 --prune-orphans 一脉相承： 凡是有破坏性或者会访问外部网络的默认行为，一律要显式 opt-in。

    brew → {duti, mas} 的链一条 hams apply --bootstrap 就能跑完 —— DAG 保证 brew 先装， brew 装好之后 brew install duti 能顺利走下去。

    macOS 上 Homebrew 的 install.sh 可能会弹出一个 Xcode Command Line Tools 的 GUI 安装对话框， 这个对话框会阻塞 stdin。如果终端看起来卡了几分钟没反应，切到桌面看看有没有一个被 IDE 压在 下面的模态窗口等着你点。

    关于 sudo

    如果你的 profile 里有 apt 这类需要提权的 provider，hams 会在启动时一次性问你密码， 然后每 4 分钟在后台自己刷 sudo 票据，保证整段 apply 过程不会再弹窗打扰你。

    失败了怎么办

    单个资源失败不会让整次 apply 都挂掉。hams 会接着跑剩下的，结束后在汇总里标出哪些失败了， 并给出退出码 4（部分失败）。日志全量保留在 ~/.local/share/hams/ 下，按月归档。 排查完再跑一次 apply，它只会重试失败的那些。

    第一次在陌生 store 上跑 apply，强烈建议先 --dry-run 看一眼清单，确认没问题再真跑。
