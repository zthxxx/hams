# 快速开始

零起步，几分钟之内就能拥有一台 hams 管起来的工作站。

## 装 hams

挑一个顺手的方式：

<Tabs items={['curl（推荐）', 'Homebrew', 'GitHub Releases']}>
  <Tabs.Tab>
```bash
bash -c "$(curl -fsSL https://github.com/zthxxx/hams/raw/main/scripts/install.sh)"
```

脚本会识别你的平台（darwin/arm64、linux/amd64、linux/arm64），下最新版的二进制放到 `/usr/local/bin/hams`，
同时记一条 channel 标记，之后 `hams self-upgrade` 就知道该从哪升。
  </Tabs.Tab>
  <Tabs.Tab>
```bash
brew install zthxxx/tap/hams
```

走的是官方 Homebrew tap。以后 `hams self-upgrade` 会自动调 `brew upgrade hams`，跟你日常的 brew 更新习惯一致。
  </Tabs.Tab>
  <Tabs.Tab>
从 [GitHub Releases](https://github.com/zthxxx/hams/releases/latest) 下自己平台的二进制：

| 平台 | 架构 | 文件 |
|------|------|------|
| macOS | Apple Silicon（M 系列） | `hams-darwin-arm64` |
| Linux | x86-64 | `hams-linux-amd64` |
| Linux | ARM64 | `hams-linux-arm64` |

```bash
# 以 macOS Apple Silicon 为例
curl -Lo hams https://github.com/zthxxx/hams/releases/latest/download/hams-darwin-arm64
chmod +x hams
sudo mv hams /usr/local/bin/hams
```
  </Tabs.Tab>
</Tabs>

装完确认一下版本：

```bash
hams --version
# hams v0.1.0 (commit: abc1234, built: 2026-04-12)
```

### Agent skill（可选，面向 Claude Code / Codex）

如果你平时用 Claude Code、Codex 之类的 agent 陪着写代码，可以顺手把 hams 自带的
skill 装上。装了之后，agent 在涉及"要把这个包装进来"或"把这台机器配起来"之类
的任务时，会更自然地去用 `hams <provider>`，而不是直接调底下的 `brew`、`pnpm`
等。对于希望长期持续维护的环境状态，这样的差别会比较明显：

```bash
npx skills add https://github.com/zthxxx/hams --skill system-packages-maintenance-hams
```

Skill 本身只是一份说明文档 —— 列清 14 个内置 provider 分别包装了什么、什么时候
用 `hams` 合适、什么时候用裸命令更合适，并指引 agent 去 `hams <command> --help`
查 flag 细节。没有运行时开销，只是让 agent 的默认行为更贴近 hams 的设计意图。

## 装点东西试试

用法很朴素 —— 把原来的命令前面加一个 `hams` 就行：

```bash
# 走 Homebrew 装 htop（比 top 好用的那个），顺手记到 Hamsfile
hams brew install htop

# 走 pnpm 全局装 serve，顺手记
hams pnpm add serve

# 配 git 身份，顺手记
hams git config --global user.name "Your Name"
hams git config --global user.email "you@example.com"
```

每条命令背后，hams 默默做两件事：

1. 真调底下的工具（`brew install htop`、`pnpm add -g serve`）
2. 把这次安装写进对应的 Hamsfile（`macOS/Homebrew.hams.yaml`、`macOS/pnpm.hams.yaml`）

<Callout type="info">
  第一次跑 `hams <provider> …` 时，hams 会帮你把 store 目录搭起来 ——
  建好 `~/.local/share/hams/store/`、跑 `git init`、写一份模板 `.gitignore` 和
  `hams.config.yaml`，同时给全局配置填上 `profile_tag: default` 和
  `machine_id: <hostname>`。你不用先跑 `hams store init`，第一条装东西的命令会顺手搞定。
</Callout>

## 在新机器上把一切恢复

这是 hams 真正发力的地方。面对一台全新的机器：

<Steps>
### 先把 hams 自己装上

```bash
bash -c "$(curl -fsSL https://github.com/zthxxx/hams/raw/main/scripts/install.sh)"
```

hams 二进制里内置了 go-git，所以哪怕这机器连 git 命令都还没有也没关系。

### 从 store 仓库 apply

```bash
# 一条命令：clone、挑 profile、apply。--tag 指明用 store 里哪一个目录
# （macOS/、openwrt/、work-linux/ 等）。--bootstrap 允许 hams 在你确认后
# 自动装前置工具。
hams apply --bootstrap --from-repo=<your-username>/hams-store --tag=macOS
```

把 `<your-username>/hams-store` 换成你自己的 GitHub 用户和 store 仓库名，
`--tag=macOS` 换成你给这类机器起的 profile 名。hams 会背着你把仓库克隆下来、
在 `~/.config/hams/hams.config.yaml` 里写好 `profile_tag: macOS` +
`machine_id: <hostname>`，然后开始 apply。

<Callout type="info">
  没带 `--tag`？交互终端下 hams 会弹出两问交互（profile tag + machine ID）让你补。
  CI / 脚本里务必显式传 `--tag=<tag>`，保证非交互。
</Callout>

**关于 bootstrap 确认。** 交互终端下 `--bootstrap` 不是必传 —— hams 跑到前置工具缺失的
provider 时会主动弹 `[y/N/s]` 让你确认。CI / 脚本里记得显式传 `--bootstrap`（或
`--no-bootstrap` 让它前置缺失时直接报错退出）。完整的 bootstrap 支持列表见
[`apply` 命令页](../cli/apply)。

### 输 sudo 密码（如果需要）

如果你的 profile 用到了需要提权的 provider（比如 Linux 上的 `apt`），hams 会在开始前一次性问密码：

```
Some providers require sudo. Enter your password to continue:
Password:
```

密码输一次就好。hams 在后台每 4 分钟刷一下 sudo 票据，保证 apply 跑到一半不会再跳出来打扰你。

### 确认一下

```bash
# 列出所有被管的资源和当前状态
```

## 把改动推回去

装完新东西记得把 Hamsfile 推回 store，其他机器才能用上：

```bash
# 站在 store 仓库目录下
hams store push
# 自动暂存所有 Hamsfile 改动 → 提交 → 推到远端
```

不想用 hams 管 git 也行，Hamsfile 是纯 YAML，`git add` + `git commit` + `git push` 随便你。
