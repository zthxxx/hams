# pnpm Provider

包装 [pnpm](https://pnpm.io/) 来管全局装的 Node.js 命令行工具 —— 像 `serve`、`tsx`、`prettier` 这种，
你在任何目录都想敲就能用的那种。

为什么专门有 pnpm provider 而不是让你用 npm？pnpm 的全局装有两点好：硬链接省磁盘、
不会把 `node_modules` 污染到系统路径里。如果你已经在用 pnpm 管项目依赖，全局也用它一致性最好。

**支持平台：** macOS、Linux

## 怎么用

```bash
# 全局装一个工具
hams pnpm add serve

# 卸掉
hams pnpm remove serve

# 看看 hams 管了哪些 pnpm 全局包
hams pnpm list
```

注意是 `add` / `remove`，这跟 pnpm 原生命令保持一致 —— 用过 pnpm 的人不用切换脑回路。

## Hamsfile 长什么样

```yaml
# macOS/pnpm.hams.yaml
schema_version: 1
provider: pnpm

groups:
  - tag: dev
    items:
      - app: serve
        intro: Static file serving and directory listing.
      - app: tsx
        intro: TypeScript executor for Node.js.
      - app: prettier
        tags: [formatter]
        intro: Opinionated code formatter.
```

`app` 就是 npm 包名，跟 `pnpm add -g <name>` 里的那个完全一致。

## 状态探测

探测方式：hams 跑 `pnpm list -g --depth=0 --json`，从 JSON 里读出全局装的包列表和版本号。
比起解析文本输出稳得多，pnpm 改输出格式也不会让 hams 突然失灵。

## 前提条件

新机器没装 pnpm？可以让 hams 帮你装。给 `apply` 加上 `--bootstrap`，hams 会在你确认后
跑 `npm install -g pnpm`：

```bash
hams apply --bootstrap --from-repo=you/hams-store
```

交互终端不带这个 flag 时，hams 会把即将执行的命令打出来让你 `[y/N/s]` 确认。非交互终端
（CI、管道）没传 flag 时，hams 直接报一条带二进制名 + 安装脚本 + `--bootstrap` 复读提示
的错误退出。详细设计见 [hams apply → 关于 bootstrap 提示](/docs/cli/apply#关于-bootstrap-提示)。

前提是机器上有 `npm` —— `npm install -g pnpm` 要跑起来得先有 node runtime，装 node 本身
是你自己的选择（nvm / fnm / n / volta / 发行版包都行，hams 不替你决定）。

想用别的 pnpm 安装方式（比如官方的 `curl -fsSL https://get.pnpm.io/install.sh | sh -`），
可以走 bash provider 一次性步骤：

```yaml
# bash.hams.yaml
- urn: "urn:hams:bash:install-pnpm"
  step: Install pnpm
  run: curl -fsSL https://get.pnpm.io/install.sh | sh -
  check: command -v pnpm
```

## 跟 npm provider 的区别

两者底层逻辑一样，只是包装的工具不同。一台机器上两个都用也没问题 —— 你愿意把 `prettier` 放 pnpm、
`nodemon` 放 npm 都行，Hamsfile 分别记录、互不打架。但一般建议选一个用：混用 node 全局包容易出现工具链诡异问题，
跟 hams 没关系，是 Node 生态本身的痛点。

<Callout type="tip">
  pnpm 全局装的东西默认放在 `~/.local/share/pnpm`，记得把它加到 `PATH`。
  `pnpm setup` 会帮你搞定这个，用 bash provider 跑一次就行。
