# Why hams?

**hams** (short for *hamster*) is a declarative IaC environment management tool for macOS and Linux workstations.
Like a hamster hoarding supplies for winter, hams hoards your environment configurations — packages, settings,
dotfiles, git repos — for safekeeping and one-command restoration on any new machine.

## The Problem

Managing a developer workstation involves hundreds of decisions made over months or years:

- `brew install ripgrep bat fd delta` because you read a blog post
- `npm install -g @anthropic-ai/claude-code` because a colleague showed you
- `defaults write NSGlobalDomain AppleShowAllExtensions -bool true` because Finder was hiding `.md` files
- `git config --global user.email "me@example.com"` on every new machine
- VS Code extensions, App Store purchases, shell configuration, symlinks...

Most developers manage this with a `setup.sh` script that grows organically into an unmaintainable
200-line bash file with no idempotency, no state tracking, no TUI feedback, and no retry logic.

**hams exists to replace that script.**

## How hams Compares

| Tool | What it does | Why it's not enough |
|------|-------------|---------------------|
| **NixOS / nix-darwin** | Fully declarative OS management via Nix expressions | Too strict — forces Nix ecosystem, no `brew install`, steep learning curve, hermetic builds break many macOS tools |
| **Terraform / Pulumi** | Declarative infrastructure as code | Cloud-first, not host-level; requires writing HCL/code *before* installing; no concept of "record what I just did" |
| **Ansible** | Playbook-based configuration management | Write-playbook-first workflow; no auto-record; re-runs all checks every time with no state file to skip unchanged resources |
| **chezmoi** | Dotfile manager with templating | Handles files only — not packages, not `defaults write`, not git config, not App Store apps |
| **brew bundle (Brewfile)** | Declarative Homebrew installs | Homebrew only; no state tracking; no comment-preserving YAML; no hooks; no cross-provider support |
| **hams** | Wraps any package manager, auto-records installations, replays on new machines | — |

## The hams Difference vs Ansible

hams and Ansible are superficially similar (both run idempotent tasks), but differ in three key ways:

1. **CLI-first auto-record** — `hams brew install git` installs *and* records. With Ansible you write a playbook first,
   then run it. With hams you install first, and the declaration is automatically generated.

2. **Explicit state file** — hams maintains a per-machine `.state/<machine-id>/` directory that tracks exactly what
   was installed and when. On the next `hams apply`, it compares desired state (Hamsfile) against observed state (`.state/`)
   and only acts on the diff. Ansible re-probes every task every run.

3. **Idempotency is explicit** — For script-type resources (bash, defaults, duti), hams requires a `check:` command
   that proves the resource is in the desired state. Ansible module idempotency is implicit and varies by module quality.

## What hams is NOT

- **Not a Docker or CI replacement.** hams configures your *host* developer environment, not ephemeral containers.
- **Not NixOS-level isolation.** hams calls real package managers (`brew`, `apt`, `pnpm`) in your real environment.
  No sandboxing, no hermetic builds, no rollback guarantees.
- **Not a project-level tool.** hams is host-level. It manages *your machine*, not a project's dependencies.
  Use `package.json` / `go.mod` / `Cargo.toml` for that.
- **Not a dotfile manager.** hams can manage dotfiles via the `file` provider, but it is not its primary focus.
  Use chezmoi or GNU Stow if you only need dotfile management.

## The Name

**hams** is short for **hamster**. Hamsters are famous for hoarding food in their cheek pouches — stuffing as much
as possible for safekeeping and retrieval later. That's exactly what this tool does: it hoards your environment
configurations into YAML files ("Hamsfiles") so you can restore everything on a new machine with a single command.

The `🐹` emoji is the official mascot.

## A Typical Day with hams

```bash
# You discover a great CLI tool. Install it and auto-record it.
hams brew install ast-grep

# Three months later, unboxing a new laptop:
hams apply --from-repo=zthxxx/hams-store
# ... hams clones your store, prompts for profile tag, installs everything

# You want to see what's managed:
hams list

# Something was installed manually outside hams. Sync state:
hams refresh
```

Your Hamsfile looks like this — hand-editable, comment-preserving YAML:

```yaml
# macOS/Homebrew.hams.yaml
schema_version: 1
provider: Homebrew

groups:
  - tag: dev-tools  # Essential development tools
    items:
      - app: git
        intro: Distributed version control system.
      - app: ripgrep
        intro: Fast recursive grep alternative.
        tags:
          - search
      - app: ast-grep   # Added by: hams brew install ast-grep
        intro: Fast code search and rewrite tool.

  - tag: media
    items:
      - app: ffmpeg
        intro: Multimedia framework for audio/video processing.
```
