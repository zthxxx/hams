5:T14eb,# 为什么要有 hams

**hams**（名字来自 *hamster*，仓鼠）是一个面向 macOS 和 Linux 工作站的声明式环境管理工具。
打个比方：仓鼠会把过冬要吃的食物塞进颊囊里囤着，hams 帮你把软件包、系统设置、dotfiles、git 仓库这些
环境配置也塞进 YAML 文件里囤着 —— 换新机器那天，一条命令全都回来。

## 这事儿的痛点在哪

一台干活用的开发机，是几个月甚至几年里成百上千个小决定攒出来的：

- 看了篇博客，顺手 `brew install ripgrep htop fd delta`
- 同事给你演示了某个工具，`npm install -g @anthropic-ai/claude-code`
- Finder 默认藏 `.md` 扩展名忍了一周，`defaults write NSGlobalDomain AppleShowAllExtensions -bool true`
- 每台新机器都得再配一次 `git config --global user.email "me@example.com"`
- VS Code 扩展、App Store 买过的应用、shell 配置、各种符号链接……

大多数人处理这些的方式是一个 `setup.sh`，从几十行慢慢长到两百行，没幂等、没状态追踪、
跑一半失败了不知道从哪续、新机器上跑完也不知道到底成功装了啥。

**hams 就是来替掉那个脚本的。**

## 跟同类工具比一下

| 工具 | 它能做什么 | 为什么还差点意思 |
|------|-----------|----------------|
| **NixOS / nix-darwin** | 用 Nix 表达式管整台机器，彻底声明式 | 太严格，必须全盘 Nix 化，不让你 `brew install`，学习曲线陡，sealed build 对很多 macOS 工具不友好 |
| **Terraform / Pulumi** | 基础设施即代码 | 面向云资源，不是本机工具；得先写 HCL 或代码才能装东西；没有"记下我刚做了什么"这个动作 |
| **Ansible** | Playbook 式的配置管理 | 先写 playbook 再跑，工作流反过来了；不能自动记录；每次都得重新探测所有任务，没有状态文件跳过没变化的资源 |
| **chezmoi** | dotfile 管理，带模板 | 只管文件 —— 软件包、`defaults write`、git config、App Store 应用一概管不了 |
| **brew bundle（Brewfile）** | 声明式跑 Homebrew | 只管 Homebrew；没状态追踪；YAML 注释不保留；没钩子；跨不了 provider |
| **hams** | 包装你已经在用的包管理器，边装边记，新机器重放 | — |

## 跟 Ansible 到底差在哪

hams 和 Ansible 表面看都能"幂等执行任务"，但有三处关键不一样：

1. **CLI 优先、自动记录** —— `hams brew install git` 同时完成了"装"和"记"。用 Ansible 你得先写个 playbook 再跑；
用 hams，你先装，声明自动落地。

2. **显式状态文件** —— hams 给每台机器维护一个 `.state/<machine-id>/` 目录，清楚记着装了啥、什么时候装的。
下次 `apply` 就拿期望状态（Hamsfile）和实际状态（`.state/`）对比，只动有差异的部分。Ansible 每次都得把所有任务重探一遍。

3. **幂等性明确** —— 对于 bash、defaults、duti 这类脚本类资源，hams 要求你明写一条 `check:` 命令来证明"这事儿已经做完了"。
Ansible 的幂等性藏在每个模块里，质量参差不齐。

## hams 不是什么

- **不是 Docker 或 CI 的替代品。** hams 管的是你的**主机**开发环境，不是临时容器。
- **不是 NixOS 那种严格隔离。** hams 直接调真实的 `brew`、`apt`、`pnpm`，在你真实的环境里动手，没有沙盒、没有 sealed build、没有万无一失的回滚。
- **不是项目级工具。** hams 管的是**你的机器**，不是项目的依赖。项目依赖用 `package.json` / `go.mod` / `Cargo.toml`。
- **不是 dotfile 管理器。** hams 可以通过 `file` provider 管 dotfiles，但不是它的主场。如果只想管 dotfiles，chezmoi 或 GNU Stow 更合适。

## 名字是怎么来的

**hams** 就是 **hamster**（仓鼠）的缩写。仓鼠的颊囊能塞很多食物，藏着过冬慢慢吃 ——
这工具做的事情一模一样：把环境配置塞进 YAML 文件（叫它们 "Hamsfile"），新机器上一条命令全掏出来。

`🐹` 是官方吉祥物。

## 日常用起来是什么样

```bash
# 发现了个好用的 CLI，装上，顺手记下来
hams brew install htop

# 三个月后，新笔记本开箱：
hams apply --bootstrap --from-repo=zthxxx/hams-store
# hams 克隆 store、问你 profile tag、自动装上缺失的前置工具（brew、pnpm、
# ansible 等，每次前有 consent 确认），然后挨个装回来。交互终端不传
# --bootstrap 也行，hams 会在每次要跑 install 脚本前弹 [y/N/s] 让你确认。

# 好奇自己管了些啥
hams list

# 手动在 hams 外面装过别的，想让状态跟现实对齐
hams refresh
```

Hamsfile 就是这样的 YAML —— 可以手动编辑，注释也不会被吃掉：

```yaml
# macOS/Homebrew.hams.yaml
schema_version: 1
provider: Homebrew

groups:
  - tag: dev-tools  # 开发基础工具
    items:
      - app: git
        intro: Distributed version control system.
      - app: ripgrep
        intro: Fast recursive grep alternative.
        tags:
          - search
      - app: ast-grep
        intro: Fast code search and rewrite tool.

  - tag: media
    items:
      - app: ffmpeg
        intro: Multimedia framework for audio/video processing.
```
