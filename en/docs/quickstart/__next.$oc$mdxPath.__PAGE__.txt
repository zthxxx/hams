# Quickstart

From zero to a fully managed workstation in a few minutes.

## Install hams

Pick whichever fits your habits:

<Tabs items={['curl (recommended)', 'Homebrew', 'GitHub Releases']}>
  <Tabs.Tab>
```bash
bash -c "$(curl -fsSL https://github.com/zthxxx/hams/raw/master/scripts/install.sh)"
```

The script figures out your platform (darwin/arm64, linux/amd64, linux/arm64), drops the latest
binary in `/usr/local/bin/hams`, and leaves a channel marker so `hams self-upgrade` knows how you
got here.
  </Tabs.Tab>
  <Tabs.Tab>
```bash
brew install zthxxx/tap/hams
```

Via the official Homebrew tap. Later, `hams self-upgrade` defers to `brew upgrade hams`, so it
fits into your regular brew habits.
  </Tabs.Tab>
  <Tabs.Tab>
Grab the binary for your platform from [GitHub Releases](https://github.com/zthxxx/hams/releases/latest):

| Platform | Architecture | File |
|----------|-------------|------|
| macOS | Apple Silicon (M-series) | `hams_darwin_arm64` |
| Linux | x86-64 | `hams_linux_amd64` |
| Linux | ARM64 | `hams_linux_arm64` |

```bash
# Example, macOS Apple Silicon
curl -Lo hams https://github.com/zthxxx/hams/releases/latest/download/hams_darwin_arm64
chmod +x hams
sudo mv hams /usr/local/bin/hams
```
  </Tabs.Tab>
</Tabs>

Confirm you're good:

```bash
hams --version
# hams v0.1.0 (commit: abc1234, built: 2026-04-12)
```

## Install something (and let hams remember)

Usage is plain: take whatever command you'd normally type, stick `hams` in front of it.

```bash
# Install htop (a nicer top) via Homebrew, and write it to the Hamsfile
hams brew install htop

# Globally install serve through pnpm, also recorded
hams pnpm add serve

# Set git identity, also recorded — natural git syntax behind `hams git`
hams git config user.name "Your Name"
hams git config user.email "you@example.com"

# VS Code extensions, also recorded
hams code install ms-python.python
```

Behind each command, hams quietly does three things:

1. Runs the actual tool (`brew install htop`, `pnpm add -g serve`, `git config --global user.name ...`, `code --install-extension ...`)
2. Writes the installation into the matching Hamsfile (`default/Homebrew.hams.yaml`, `default/pnpm.hams.yaml`, `default/git-config.hams.yaml`, `default/code.hams.yaml`)
3. **Auto-creates the store** at `~/.local/share/hams/store/` on first run if you don't have one yet — `git init`'d, with a bundled `.gitignore` and `hams.config.yaml`. No `hams store init` first.

Set `HAMS_NO_AUTO_INIT=1` if you want the legacy "no store directory configured" hard-fail (e.g., in CI scripts that pre-provision the store).

## Restore everything on a fresh machine

This is where hams earns its keep. On a brand-new machine:

<Steps>
### First, install hams itself

```bash
bash -c "$(curl -fsSL https://github.com/zthxxx/hams/raw/master/scripts/install.sh)"
```

hams bundles go-git, so you don't need to install `git` first. Classic chicken-and-egg dodged.

### Apply from your store repo

```bash
# One-command restore: clone the store, pick the profile, run.
# Add --bootstrap if the machine doesn't have Homebrew / pnpm / ansible etc.
# yet so hams auto-installs prerequisites under your consent.
hams apply --from-repo=<your-username>/hams-store --tag=macOS

# Or with bootstrap allowed:
hams apply --bootstrap --from-repo=<your-username>/hams-store --tag=macOS
```

Replace the placeholder with your GitHub username and store repo name. On an
interactive terminal, `--bootstrap` is optional — hams will show a `[y/N/s]`
prompt before running any install script. In CI / scripts, pass `--bootstrap`
explicitly (or `--no-bootstrap` to fail fast).

### No interactive prompts needed

If `~/.config/hams/hams.config.yaml` doesn't exist yet, hams writes a default one
on first run with `tag: default` and a hostname-derived `machine_id`. Pass
`--tag=<your-profile>` to override the default for the current run; the value also
gets persisted via `hams config set tag <name>` for future runs.

The legacy interactive `Profile tag / Machine ID` prompt is reserved for `hams
store init`, which you only need when you're hand-crafting a store outside the
default `${HAMS_DATA_HOME}/store/` location.

### Enter your sudo password, if anything needs it

If your profile uses a privileged provider (like `apt` on Linux), hams asks for sudo once at the start:

```
Some providers require sudo. Enter your password to continue:
Password:
```

Once is all it needs. hams keeps the sudo ticket warm in the background (refreshing every 4 minutes)
so apply can run to completion without popping another prompt.

## Push your changes back

Once you've installed new things on this machine, push the Hamsfiles back to the store so the
other machines can catch up:

```bash
# From inside your store directory
hams store push
# Stages Hamsfile changes, commits, pushes to the remote
```

Prefer to drive git yourself? Go for it. Hamsfiles are plain YAML, nothing hams does stops you
from `git add && git commit && git push`.
